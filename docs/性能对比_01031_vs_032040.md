# 01-031 流式状态机 vs 032-040 独立函数扫描：性能对比

本文比较的是当时工程里的两种实现形态：

- `01-031`：批量状态机构建 + 批量导出
- `032-040`：独立函数逐个扫描 `cross_section`

这不是完全同构的实现对比，因此结论只在明确场景下成立。

补充：

- 该文档反映的是改造前的比较背景
- 当前代码已经补上 `collect_factor032_040(...)` 批量入口
- 因此本文中关于 `032-040` “逐个独立扫描”的描述，现仅适用于旧版 wrapper 路径的背景说明

## 核心区别

| 维度 | 01-031 (流式状态机) | 032-040 (独立函数扫描) |
|------|---------------------|------------------------|
| **扫描次数** | 对 order/trade/cross_section 各 **1 遍** | 每个因子 **独立扫描** cross_section 1 遍 |
| **状态共享** | 31 个因子共享一份 `Factor01_031State` | 9 个因子之间 **无共享状态** |
| **临时分配** | 分配集中在 `unordered_map`、quantile 样本和工作区 | 每个因子调用都可能构造临时 `vector`（如 `calc_diff_returns_in_window` 返回 `vector<double>`） |
| **架构复杂度** | 三层（state → stream → extract），~300 行状态定义 | 扁平的 utility 函数 + 直调，整体 ~100 行 |

## 批量计算场景（生产环境的主场景）

这是实际使用的模式——在 `predict_t` 时刻一次性算完全部因子。

在“`predict_t` 时刻一次性导出整组因子”的前提下，`01-031` 的当前实现更有优势：

- `collect_factor001_031(input)` 只做 **1 次** order 扫描 + **1 次** trade 扫描 + **1 次** cross_section 扫描 → 导出全部 31 个值
- 时间复杂度可写成：O(n_order + n_trade + n_cs)
- 该优势来自“共享中间状态”，而不是单个因子的常数项一定更小

相比之下，`032-040` 当前实现存在明显冗余扫描：

- 9 个 `call_factorXXX()` 各自独立遍历 `cross_section`
- factor036 和 factor037 都调用 `calc_diff_returns_in_window()` 分别构造 `vector<double>` → 重复计算 + 重复分配
- 时间复杂度：O(9 × n_cs)，实际常数还会叠加多次 `vector` 分配和拷贝

具体冗余：

| 因子 | 扫描函数 | 重复情况 |
|------|----------|----------|
| 034, 035 | `count_cross_section_in_window` | 扫描 2 次相同窗口 |
| 036, 037 | `calc_diff_returns_in_window` | 各自构造一份 diff_returns vector |
| 038, 039 | `max_bid_amount_with_time_in_window` | 扫描 2 次 + 039 还需 `last_bid_amount_in_window` 第 3 次 |
| 040 | `collect_bid_amounts_in_window` | 又扫描 1 次并构造 vector |

## 单因子查询场景

当前实现下，单因子查询时 `032-040` 更有优势：只扫描一次 `cross_section`，只算需要的那个因子。

当前实现下，单因子查询时 `01-031` 有明显浪费：每个 `call_factorXXX()` 都会调用 `build_factor01_031_state()` 构建完整状态（含 `unordered_map` 初始化、全量 `order/trade` 扫描），即使只需一个标量结果。

## 内存和 Cache 效率

| 指标 | 01-031 | 032-040 |
|------|--------|---------|
| 堆分配次数（批量） | `unordered_map` 桶/节点分配 + quantile samples 追加 + workspace 拷贝 | 主要来自若干临时 `vector` 构造 |
| Cache 友好性 | 较优：顺序扫描线性数据，状态可复用 | 中：重复扫描同一数组，且部分因子会生成临时 `vector` |
| 峰值内存 | `unordered_map` 大小取决于唯一 `order_id` 数量 | 临时 `vector` 峰值 ≈ 窗口内 cross_section 条数 |

## 结论

更准确的结论是：

- 在当前实现下，若场景是“批量一次性计算整组因子”，`01-031` 更有优势
- 在当前实现下，若场景是“只查单个因子”，`032-040` 更轻
- `01-031` 的核心优势是 **"一次构建状态，多次导出因子"**
- 这个优势来自状态共享，不应外推为“任意场景下都更快”

补充说明：

- 如果 `n_order + n_trade` 远大于 `n_cs`，则不能只凭 Big-O 形式断言 `01-031` 一定快于 `032-040`
- 反过来，如果真实生产路径本来就必须顺序消费 `order/trade` 流，那 `01-031` 的状态机构建成本往往是必付成本

## 优化建议：将 032-040 升级为批量模式

参照 01-031 的模式，增加一个批量入口：

```cpp
struct Factor032_040State {
    std::size_t window_count;
    int         window_duration;
    double      var_95;
    std::vector<double> diff_returns;  // 只构造一次
    double      max_bid_amount;
    int         max_bid_time;
    double      last_bid_amount;
    std::vector<double> bid_amounts;   // 只构造一次
};

Factor032_040Result collect_factor032_040(const FactorInput& input);
```

这样 9 个因子只需 **1 次** `cross_section` 扫描，比较口径才会更接近 `01-031` 的批量模式。
