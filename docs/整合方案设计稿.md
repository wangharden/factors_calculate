# 因子代码整合方案设计稿

## 1. 背景与动机

### 当前文件现状

```
factor.h                  — 输入数据结构
cross_section_utils.h     — 公共工具（时间/统计/截面扫描）
factor01_031_state.h      — 001-031 + 041-050 的状态/结果结构
factor01_031_stream.h     — 001-031 + 041-050 的流式状态构建
factor01_031_extract.h    — 001-031 + 041-050 的因子提取
factor032_040_extract.h   — 032-040 的状态、构建、提取（all-in-one）
trade_vwap_utils.h        — 053-061 的状态、构建、提取（all-in-one）
factor_collection.h       — 统一入口薄封装
```

### 存在的问题

| 问题 | 描述 |
|------|------|
| **Result 结构碎片化** | 四个独立 Result 类型分散在三个命名空间中：`factor01_031_detail::Factor01_031Result`、`factor01_031_detail::Factor041_050Result`、`factor032_040_detail::Factor032_040Result`、`factor_vwap_detail::VwapRatioFactorResult`。调用方需要分四次调用、持有四个结构。 |
| **factor_collection.h 过长** | 单因子 `call_factorXXX()` 封装共 50 个函数，占 ~350 行，且每个都完整重建 state。 |
| **缺少全量入口** | 没有一个函数能一次性计算全部 50 个因子并返回统一结构。 |

### 整合目标

1. **统一 Result 结构**：提供一个 `AllFactorResult` 供撤单决策一次性拿到全部 50 个因子值
2. **新增全量入口**：`collect_all_factors(input)` 一次调用得到全部结果
3. **保留分组批量入口**：`collect_factor001_031()` 等不删除，仍可独立使用
4. **保持 header-only inline 风格**
5. **不改变因子计算逻辑**，纯重构

## 2. 整合方案总览

### 2.1 041-050 保留在 01-031 文件中

041-050 和 001-031 共享同一份 `Factor01_031State` 和同一遍 order/trade 扫描。状态构建、因子提取都高度耦合，没有拆分收益。保持现状：

```
factor01_031_state.h    → Factor01_031State + Factor01_031Result + Factor041_050Result（不变）
factor01_031_stream.h   → 001-031 + 041-050 的流式状态构建（不变）
factor01_031_extract.h  → 001-031 + 041-050 的因子提取（不变）
```

### 2.2 统一 Result 结构

新增 `factor_result.h`：

```cpp
#ifndef LIMIT_UP_FACTOR_DEMO_FACTOR_RESULT_H
#define LIMIT_UP_FACTOR_DEMO_FACTOR_RESULT_H

struct AllFactorResult {
    double factor001;
    double factor002;
    // ...
    double factor050;
    double factor053;
    // ...
    double factor061;
};

#endif
```

**语义说明**：

- `AllFactorResult` 的 50 个字段按编号连续排列：001-031（31 个）、032-040（9 个）、041-050（10 个）、053-061（9 个），不留 051/052 的空位
- 当前阶段把它作为**工程内部的强类型结果对象**
- 各分组 Result 保留不删除，方便单独调试/测试

### 2.3 factor_collection.h 精简

当前 `factor_collection.h` 有三层职责：

1. 分组批量入口转发（`collect_factor001_031` 等）
2. 50 个 `call_factorXXX()` 单因子封装
3. `AllFactorResult` 全量计算入口（待新增）

**调整方案**：

```
factor_collection.h    → 保留分组批量入口 + 新增 collect_all_factors()
                         删除 50 个 call_factorXXX()（移到 factor_single_call.h 或直接废弃）
```

新增核心函数：

```cpp
inline AllFactorResult collect_all_factors(const FactorInput& input) {
    AllFactorResult all;

    // 第一遍：order + trade + cross_section → Factor01_031State
    factor01_031_detail::Factor01_031State state_01_031;
    factor01_031_detail::build_factor01_031_state(input, state_01_031);
    factor01_031_detail::Factor01_031Workspace workspace_01_031;

    // 导出 001-031
    Factor01_031Result r1 = factor01_031_detail::collect_factor001_031(
        input, state_01_031, workspace_01_031);
    // 复用同一 state 导出 041-050
    Factor041_050Result r4 = factor01_031_detail::collect_factor041_050(
        input, state_01_031, workspace_01_031);

    // 第二遍：cross_section → Factor032_040State
    factor032_040_detail::Factor032_040State state_032_040;
    factor032_040_detail::build_factor032_040_state(input, state_032_040);
    factor032_040_detail::Factor032_040Workspace workspace_032_040;
    Factor032_040Result r3 = factor032_040_detail::collect_factor032_040(
        input, state_032_040, workspace_032_040);

    // 第三遍：trade → VwapRatioStatsState
    factor_vwap_detail::VwapRatioStatsState vwap_state;
    factor_vwap_detail::build_vwap_ratio_stats(input, vwap_state);
    VwapRatioFactorResult r5 = factor_vwap_detail::collect_factor053_061(vwap_state);

    // 显式字段赋值
    all.factor001 = r1.factor001;
    // ...
    all.factor061 = r5.factor061;
    return all;
}
```

**性能特征**：

| 扫描 | 数据 | 次数 | 产出 |
|------|------|------|------|
| order + trade | `build_factor01_031_state` | 1 | 001-031 + 041-050 |
| cross_section | `build_factor032_040_state` | 1 | 032-040 |
| trade | `build_vwap_ratio_stats` | 1 | 053-061 |

当前实现形态下，`collect_all_factors()` 会做 3 次分组批量构建。

备注：

- `041-050` 已接入 `collect_all_factors()`，并复用 `001-031` 的同一份状态
- 未来是否进一步合并扫描，取决于输入组织方式和后续优化优先级，不在本稿当前范围内

### 2.4 单因子调用的处理

50 个 `call_factorXXX()` 函数当前每个都完整重建 state，在实际使用中几乎不会单独调用（因为总是一次算完全部因子）。

**建议**：

- **方案 A（推荐）**：移到一个单独的 `factor_single_call.h`，不在主路径上 include，仅供调试和测试使用
- **方案 B**：直接从 `factor_collection.h` 删除，需要单因子时直接调 `collect_all_factors()` 后取对应字段

```

factor_result.h            — 【新】AllFactorResult
```

### 与当前文件的差异一览

## 4. 执行步骤

按依赖顺序，建议分两步：

### 步骤 1：新增 `factor_result.h`

- 新增 `AllFactorResult`
- 各分组 Result 不动，留在原位

### 步骤 2：整合 `factor_collection.h`

- 新增 `collect_all_factors()` 函数
- 将 50 个 `call_factorXXX()` 移到 `factor_single_call.h`（可选）
- 更新 CMakeLists.txt

## 6. 调用方视角

### 整合前（当前）

```cpp
#include "factor_collection.h"
#include "trade_vwap_utils.h"

// 调用方需要拿 4 个不同类型的结果
auto r1 = collect_factor001_031(input);
auto r3 = collect_factor032_040(input);    // 重复扫描 cross_section
auto r4 = collect_factor041_050(input);    // 重复扫描 order + trade
auto r5 = build_and_collect_vwap(input);   // 需要手动构建 vwap_state

// 传给决策函数要传 4 个参数
decide(r1, r3, r4, r5);
```

### 整合后

```cpp
#include "factor_collection.h"

// 一次调用，全部因子
AllFactorResult all = collect_all_factors(input);

// 传给决策函数
decide(all);

```

### 仍支持分组调用（不破坏已有代码）

```cpp
// 这些接口全部保留
auto r1 = collect_factor001_031(input);
auto r3 = collect_factor032_040(input);
```

## 6. 风险与注意事项

| 风险 | 缓解 |
|------|------|
| `AllFactorResult` 中 051/052 不存在，数组下标与因子编号不连续 | 文档明确 50 个字段的编号映射关系；不予空位填充（避免永久浪费） |

## 7. 未来优化方向

当前方案先不把 `AllFactorResult` 绑定为“紧凑内存布局协议”。

后续如果决策侧明确需要：

- `const double*` 数组视图
- 无拷贝顺序访问
- 与外部数值模块直接对接

可以再评估增加：

- `as_array(const AllFactorResult&)`
- 导出到 `double out[50]` 的适配函数
- 更激进的紧凑布局 / `memcpy` 组装方案

这些都属于未来优化，不作为当前重构前置条件。
