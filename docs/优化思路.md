# 代码落地原则

## 1. CrossSection 主表示

- `cross_section` 当前主表示固定为 AoS：`std::vector<FactorCrossSectionInfo>`
- 在未确认 SIMD 热点前，不为 `CrossSectionSoA` 引入双存储和同步成本
- 只有在 profiling 证明 `cross_section` 是热点，且瓶颈来自单列批处理时，才允许把 `CrossSectionSoA` 接入主执行路径

## 2. 原始数据只读

- 原始 `cross_section / order / trade` 输入一律视为只读
- 因子计算阶段不允许对原始行情数据做排序、重排或覆盖写入
- 任何会改变元素顺序的操作，只能作用于临时工作区

## 3. 公共函数设计

- `cross_section` 公共函数默认采用顺序扫描实现
- 凡是能在一次遍历中完成的窗口过滤、计数、峰值、末值提取，不先构造窗口子序列
- 因子函数不得重复拼装窗口逻辑，窗口计数、精确时刻摘要、峰值扫描、差分收益率等能力统一下沉到公共工具函数

## 4. 统计与分位数

- `mean / std / skew` 优先接受只读输入并顺序扫描完成
- `quantile / median` 与 pandas `Series.quantile(..., interpolation="linear")` 对齐，采用线性插值
- 单分位数优先使用选择算法，不默认全量排序
- 只有在确实需要完整有序序列时，才允许使用 `sort`
- `quantile_in_place` 的语义是“对工作区原地选择并返回线性插值分位数”，不允许直接传入原始输入容器

## 5. 拷贝与工作区

- 任何新增临时 `vector` 都要有理由
- 只有在算法必须保留整段样本时，才允许分配中间数组
- 可以使用工作区复制原始数据后做重排，但应优先复用工作区，避免重复分配
- 代码评审时，对 `cross_section` 工具层重点检查两件事：
  - 是否符合当前 AoS 访问模式
  - 是否引入了不必要的中间拷贝

## 6. 当前阶段边界

- 涉及 `order / trade` 全量事件序列的公共层不在本阶段实现
- 当前阶段优先完善 `cross_section` 的基础层和统计工具，而不是提前扩展输入面
